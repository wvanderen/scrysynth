## Why Plan 11-02 is paused

**Structural gap discovered during UAT scenario 7 (agent approval):** the production conversation path cannot drive the planner / pending-action / approval flow.

### Root cause

`src-tauri/src/lib.rs:113-133` — `send_agent_message` calls `agent_command::parse_agent_intent` directly and feeds the result to `apply_agent_command`. It does **not** invoke `plan_and_apply_agent_request` (`src-tauri/src/application/agent_command.rs:362`), which is the function Phase 10 verified through Rust integration tests.

Consequences:
- The `PlannerProvider` machinery (`agent_planner.rs`) is unreachable from the GUI.
- The local regex parser never emits a high-risk command, so no production GUI flow ever creates a pending action.
- The `PendingActionCard` UI in `ConversationView.tsx` is rendered but has no production data path feeding it.
- The fixtures Phase 10 UAT exercised (`mock-fixture-planner`, `json-test-provider`, `local-parser`) live in `src-tauri/tests/agent_commands.rs`, not in the production code path.

**This is not a Plan 11-01 regression.** Phase 10's UAT was technically honest (`"does not verify... GUI click-through in the packaged desktop app"`), but that caveat was not propagated to REL-02 scope. Phase 11 inherited the gap.

### Secondary issue (visual runtime)

Operator reported "visual runtime doesn't seem to be starting up." Initial characterization: the visual runtime does **not** auto-start — the user must switch to the runtime workspace tab and click the Visual Runtime card's Start button. The minimal sidecar has no visible window; running state is only observable via Runtime Health status going `idle` → `ready`. Re-confirm after the planner fix whether scenario 8 (visual runtime) still passes; if Start was clicked and failed silently, capture `visualRuntime.status`, `visualRuntime.detail`, and `visualRuntime.connectionStatus` from the panel for diagnosis.

## Fix scope (new GSD task)

The fix is more than a one-line wire-up. The next GSD session should:

1. **Wire the planner into `send_agent_message`.** Replace the direct `parse_agent_intent` + `apply_agent_command` call with `plan_and_apply_agent_request` using a default provider. `ParserPlannerProvider::default()` is the natural default (it already wraps `parse_agent_intent`) — preserves current behavior while routing through the planner gates.
2. **Ensure at least one user-typed input produces a high-risk pending action.** The local parser today never emits a high-risk command; either extend the parser to classify certain agent-attributed commands (e.g. `remove <agent-owned node>`) as high-risk, or add a deterministic v1 provider that does. Without this, scenario 7 still cannot be verified even after the wiring fix.
3. **Add a Rust integration test** that drives `send_agent_message` end-to-end and asserts a pending action appears for the chosen trigger input.
4. **Re-run Phase 10 UAT** to confirm no regression in the existing planner path.
5. **Rebuild the .app** (`npm run tauri build`) so Plan 11-02 UAT can re-target the fixed packaged artifact.

Suggested routing: `/gsd:debug` (this is an investigation + fix + test cycle) or `/gsd:quick` if the scope feels small. Either way, the new session owns the code change; Plan 11-02 then resumes from Task 1 step 6.
